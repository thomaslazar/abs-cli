# `me` + `items progress` + `items get --include` — design

Status: drafted 2026-05-30. Targets v0.6.0.

## Summary

Bundle of three additions to the CLI's `/api/me/*` surface plus one
modification to `items get`:

```
me

items progress get        --library-item <lid>
items progress set        --library-item <lid> [body flags…]
items progress remove     --library-item <lid>
items batch-update-progress (--input <file>|--stdin)

items get --id <id> [--expanded] [--include <flags>]
```

7 new verbs, 1 modified verb. Shipping as one PR because the surface is
cohesive: `--include progress` reads what `items progress set` writes,
and `me` shows whose progress the writes will affect. See research:
[docs/specs/research/2026-05-30-me-progress-include.md](research/2026-05-30-me-progress-include.md).

## Motivation

The CLI today has no way to:

- See who is authenticated. `config get` shows server + masked tokens
  only; the username is printed once at login and unrecoverable without
  decoding the token.
- Mark a book as listened / read / in-progress. Agents can list and
  modify books, but cannot say "this one is done" so the user's view
  isn't perpetually claiming everything is unstarted.
- Inspect the user's progress on an item without a second
  `/api/me/progress/...` call. ABS supports `?include=progress` on
  `items get` but only when `expanded=1` — the CLI hasn't exposed
  either part of that pair.

This bundle closes all three. Permission model is minimal: every
endpoint requires only authentication (each user manages their own
progress), so no `AddPermissionRequired` tags.

## ABS endpoints touched

| CLI verb | Method | Path | Body | Permission |
|---|---|---|---|---|
| `me` | GET | `/api/me` | – | (auth) |
| `items progress get` | GET | `/api/me/progress/:libraryItemId` | – | (auth) |
| `items progress set` | PATCH | `/api/me/progress/:libraryItemId` | partial fields | (auth) |
| `items progress remove` | GET → DELETE | `/api/me/progress/:lid` → `/api/me/progress/:progressId` | – | (auth) |
| `items batch-update-progress` | PATCH | `/api/me/progress/batch/update` | JSON array | (auth) |
| `items get --include` | GET | `/api/items/:id?expanded=1&include=<flags>` | – | (auth + library access) |

Routing: `temp/audiobookshelf/server/routers/ApiRouter.js:173-181`
(me + progress) and `:104` (items get).

### Server-side asymmetry: `progress remove`

ABS's `DELETE /api/me/progress/:id` takes the `MediaProgress` row's
`id`, not the `libraryItemId`. Every other progress endpoint takes the
`libraryItemId`. To keep the CLI surface symmetric, `items progress
remove --library-item <lid>` internally:

1. GET `/api/me/progress/:libraryItemId` to discover the row's `id`.
2. DELETE `/api/me/progress/:id`.
3. Print `{"success":"true"}` on 200.

Two HTTP calls behind one verb. Justification: the rule
`feedback_cli_thin_passthrough` exists to prevent hidden interpretation
and smart-defaulting. Here the extra call is a pure server-side
parameter-naming asymmetry workaround — atomic from the caller's view,
no decision points between the two calls, no values inferred or
modified. The alternative (`--progress-id`) would force every caller to
GET the progress first themselves, which is the same two calls with
worse ergonomics.

### Server-side asymmetry: `items get --include` requires `expanded=1`

`LibraryItemController.findOne` (`LibraryItemController.js:60-101`)
parses `include` only inside the `if (req.query.expanded == 1)` block.
The minified path silently ignores `include`. The CLI auto-implies
`expanded=1` when `--include` is passed; `--help` documents the
behavior.

### Progress payload

`PATCH /api/me/progress/:libraryItemId` body fields, with semantics from
`temp/audiobookshelf/server/models/User.js:67-81` and
`MediaProgress.js:155-176`:

| Field | Type | Meaning |
|---|---|---|
| `currentTime` | number (s) | Audio position in seconds |
| `isFinished` | bool | "Listened/read" marker |
| `ebookLocation` | string | EPUB CFI (or empty string to clear) |
| `ebookProgress` | number (0..1) | Ebook completion fraction |
| `hideFromContinueListening` | bool | Hide from the "continue" feed |
| `finishedAt` | number (ms) | Finish timestamp |

Fields the CLI deliberately does NOT expose: `lastUpdate` (server
manages), `duration` (server derives), `markAsFinishedTimeRemaining` /
`markAsFinishedPercentComplete` (server-internal preferences).

### MediaProgress JSON (response shape)

`MediaProgress.getOldMediaProgress()`:

```
{
  id, userId, libraryItemId, episodeId, mediaItemId, mediaItemType,
  duration, progress,
  currentTime, isFinished,
  ebookLocation, ebookProgress,
  hideFromContinueListening,
  lastUpdate, startedAt, finishedAt
}
```

Single record per `(user, libraryItem)` — both audio and ebook fields
live on the same row. `progress` is the derived `currentTime / duration`
fraction.

### `/api/me` JSON (response shape)

`User.toOldJSONForBrowser()` — see research doc for the full breakdown.
Includes the FULL `mediaProgress` array, which can be MB-size on power
users. Server has no slimming param exposed on this endpoint.

## Command surface

### `me`

```
abs-cli me
```

No flags. Prints the `Me` response (the full user object) on stdout.

### `items progress get`

```
abs-cli items progress get --library-item <lid>
```

| Flag | Required | Notes |
|---|---|---|
| `--library-item` | yes | The library item ID (not the progress record ID) |

Returns the single MediaProgress row. 404 surfaces as "Not found" exit
2 if no progress exists for this item.

### `items progress set`

```
abs-cli items progress set --library-item <lid>
    [--current-time <seconds>]
    [--is-finished <true|false>]
    [--ebook-location <cfi>]
    [--ebook-progress <0..1>]
    [--hide-from-continue-listening <true|false>]
    [--finished-at <iso8601>]
```

| Flag | Required | Notes |
|---|---|---|
| `--library-item` | yes | Library item ID |
| `--current-time` | no | Audio position in seconds; floats allowed |
| `--is-finished` | no | `true` or `false` (required value — see below) |
| `--ebook-location` | no | EPUB CFI string. Empty string `""` clears |
| `--ebook-progress` | no | 0..1 number |
| `--hide-from-continue-listening` | no | `true` or `false` |
| `--finished-at` | no | ISO 8601; CLI converts to ms-since-epoch |

At least one body flag is required. Booleans take **explicit `true|false`**
rather than presence-only because the wire semantics need ternary
(`unchanged | true | false`) — a presence flag can't express "set to
false." `Option<bool?>` with required argument when the flag appears.

Server returns empty 200. CLI does a **follow-up GET** to
`/api/me/progress/:lid` and prints the resulting MediaProgress — same
echo-the-post-state pattern as `items update`. Two HTTP calls behind
one verb, same justification as `progress remove`.

### `items progress remove`

```
abs-cli items progress remove --library-item <lid>
```

| Flag | Required | Notes |
|---|---|---|
| `--library-item` | yes | Library item ID |

Internal GET → DELETE (see "Server-side asymmetry" above). Prints
`{"success":"true"}` on 200.

`--help` blurb:

> Removes both audio and ebook progress in one shot — server has no
> per-half delete. To reset only one half, use `items progress set`
> with cleared values for the half you want zeroed (e.g.
> `--ebook-location "" --ebook-progress 0`).

### `items batch-update-progress`

```
abs-cli items batch-update-progress (--input <file>|--stdin)
```

| Flag | Required | Notes |
|---|---|---|
| `--input` | yes\* | JSON file with array body |
| `--stdin` | yes\* | Read JSON array from stdin |

\* Exactly one of `--input` or `--stdin`.

Body is a JSON array of partial progress payloads, each containing
`libraryItemId` plus any subset of the PATCH fields:

```json
[
  {"libraryItemId":"li_abc","isFinished":true},
  {"libraryItemId":"li_def","currentTime":1234.5}
]
```

Raw passthrough to `PATCH /api/me/progress/batch/update`. Prints
`{"success":"true"}` on 200.

`--help` blurb:

> Server returns 200 even when individual entries fail (errors only
> logged server-side, no per-entry feedback in the response). Recommend
> pre-validating client-side or following up with `items progress get`
> for entries that matter.

### `items get --include` (modifies existing verb)

```
abs-cli items get --id <id> [--expanded] [--include <flags>]
```

New flag: `--include`. Comma-separated subset of
`progress,rssfeed,share,downloads`. Examples:

```
abs-cli items get --id <id> --include progress
abs-cli items get --id <id> --include progress,rssfeed
abs-cli items get --id <id> --expanded --include progress
```

Semantics:
- Passing `--include` automatically sets `expanded=1` on the wire,
  whether or not `--expanded` was passed.
- Both `--expanded` and `--include` may be supplied together (no
  conflict).
- Comma-separated values are sent verbatim to ABS (server lower-cases
  and splits).

`--help` blurb:

> `--include` requires the expanded item shape — passing `--include`
> automatically implies `--expanded`. Values: `progress` (your media
> progress for this item), `rssfeed` (open RSS feed if any), `share`
> (admin and book-only; silently skipped otherwise), `downloads`
> (podcast-only; silently skipped for books).

### New: `src/AbsCli/Services/ProgressService.cs`

```csharp
public class ProgressService
{
    private readonly AbsApiClient _client;
    public ProgressService(AbsApiClient client) { _client = client; }

    public async Task<MediaProgress> GetAsync(string libraryItemId)
        => await _client.GetAsync(ApiEndpoints.MeProgress(libraryItemId),
                                  AppJsonContext.Default.MediaProgress);

    public async Task SetAsync(string libraryItemId, ProgressUpdateRequest body)
    {
        var json = JsonSerializer.Serialize(body, AppJsonContext.Default.ProgressUpdateRequest);
        await _client.PatchAsync(ApiEndpoints.MeProgress(libraryItemId), json);
    }

    public async Task RemoveAsync(string progressId)
        => await _client.DeleteAsync(ApiEndpoints.MeProgressById(progressId));

    public async Task BatchUpdateAsync(string jsonBody)
        => await _client.PatchAsync(ApiEndpoints.MeProgressBatchUpdate, jsonBody);
}
```

`SetAsync` uses a typed `ProgressUpdateRequest` DTO with
`JsonIgnore(WhenWritingNull)` on every nullable field. This sidesteps
the `Dictionary<string, object?>` source-gen incompatibility (the
codebase has no precedent for polymorphic-value dictionaries — every
existing dictionary serializer is `Dictionary<string, string>`), gives
agents field discoverability via `--help`, and keeps the PATCH body
omit-vs-set semantics clean per field.

## AOT considerations

All new DTOs registered via `[JsonSerializable]`. The progress PATCH
body has tri-state semantics per field (omit = unchanged, value = set);
the typed `ProgressUpdateRequest` with `JsonIgnore(WhenWritingNull)`
handles this cleanly — no need for the `Dictionary<string, string>` /
`null!` pattern used by `authors update`, since booleans and numbers
can't use the empty-string sentinel.

If `Dictionary<string, object?>` is preferred for `SetAsync`, verify
source-gen compatibility before committing to it.

The batch endpoint passes through a raw JSON string from
`--input`/`--stdin` — no DTO needed for the array container.

## Error handling

Standard CLI error model:

| Failure | Exit | stderr |
|---|---|---|
| 401 / 403 | 2 | Standard `EnsureValidTokenAsync` flow or "Permission denied" (no `--include share` from non-admin) |
| 404 on `progress get` (no progress exists) | 2 | `Not found. <server message>` |
| 404 on `progress remove` (no progress to remove) | 2 | Surfaces from the internal GET — same message |
| Malformed `--input` JSON | 1 | `Invalid JSON input: <parse error>` |
| Both `--input` and `--stdin` provided | 1 | `Provide --input or --stdin, not both.` |
| Neither provided where required | 1 | `Provide --input <file> or --stdin.` |
| No body flags on `progress set` | 1 | `Specify at least one of --current-time, --is-finished, ...` |
| Invalid `--is-finished` value (not `true`/`false`) | 1 | System.CommandLine's native parser error |

## Sharp edges to verify during implementation

- **`Bookmarks` and `MediaItemShare` shape**: server returns these but
  the CLI doesn't model them in detail. Use `JsonElement` / nullable
  list of `JsonElement` to round-trip without committing to a schema
  we'd have to maintain. `Me.Bookmarks` is `List<JsonElement>`;
  `LibraryItemExpanded.MediaItemShare` is `JsonElement?`.

## Out of scope (deferred)

- **Podcast episodes.** All progress endpoints accept an optional
  `:episodeId?` for podcasts. Adding `--episode` is additive; punt
  until podcast support arrives (currently in `docs/roadmap.md`'s
  Deferred features table).
- **Bookmarks.** Separate ABS endpoints (`/api/me/item/:id/bookmark`).
  Out of scope for this milestone; round-tripped in the `Me` response
  as raw `JsonElement`.
- **Listening sessions / stats.** `/api/me/listening-sessions`,
  `/api/me/listening-stats`, `/api/me/items-in-progress`,
  `/api/me/stats/year/:year` — read-only history endpoints, not
  required by the agent metadata workflow. Future milestone.
- **`/api/me/password`** — credential change. Out of scope; the CLI
  isn't an admin tool for user management.
- **Admin "lookup other user"** — `/api/me` strictly returns the
  caller. Admin lookups go through `/api/users/:id`, a separate
  resource not in this milestone.

## Open questions

- **`Dictionary<string, object?>` vs typed request DTO for
  `SetAsync`** — resolved during implementation. Lean: typed
  `ProgressUpdateRequest` for AOT compatibility + field
  discoverability via `--help`.
- **Should `--include` validate values client-side?** Lean: no. Server
  is permissive (lowercases + splits; unknown values become no-ops),
  and over-strict client validation would block valid future ABS
  additions. Document the four known values in `--help` without
  rejecting others.
- **`--finished-at` accepts ISO only or also raw ms?** Lean: ISO only.
  ms timestamps are awkward to type and the CLI's other timestamp
  fields (when they exist) use ISO. Document in `--help`.

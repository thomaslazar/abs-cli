# `items encode-m4b` — design

Status: approved 2026-05-12. Targets v0.5.0.

## Summary

Add two leaf commands under `items encode-m4b` that wrap ABS's
"Encode M4B" tool — merging a multi-file audiobook into a single
tagged `.m4b`:

```
abs-cli items encode-m4b start  --id <id> [--codec ...] [--bitrate ...] [--channels ...]
abs-cli items encode-m4b cancel --id <id>
```

Both endpoints are admin-only. The commands are a deliberate
fire-and-forget primitive — no `--wait` / no progress polling baked
into the CLI. Observability is documented (`tasks list` for in-flight,
`items get` for outcome) but not papered over with mechanisms we know
to be racy on ABS's side.

## Motivation

Agents fixing up multi-file audiobooks (post-import, post-metadata
cleanup, before handing off to external players) want to consolidate
into a single tagged `.m4b`. ABS's web UI exposes this under
**Tools → Encode M4B**; the CLI does not. The result on disk is what a
human would otherwise click-through to produce: original tracks moved
to ABS's metadata cache as a backup, a single new `.m4b` written to
the item's library directory with chapters + cover embedded.

This is the first of four v0.5.0 "file management" primitives.

## ABS endpoints touched

| CLI verb | Endpoint | Method | Body | Notes |
|----------|----------|--------|------|-------|
| `start` | `/api/tools/item/:id/encode-m4b` | POST | empty | Query params: `codec`, `channels`, `bitrate`. Returns `200` empty. |
| `cancel` | `/api/tools/item/:id/encode-m4b` | DELETE | empty | Returns `200` empty on success; `404` if no pending task or item not found. |

Routes confirmed at
`temp/audiobookshelf/server/routers/ApiRouter.js:297-298`. Controllers:
`temp/audiobookshelf/server/controllers/ToolsController.js:29`
(`encodeM4b`) and `:66` (`cancelM4bEncode`). The `ToolsController`
middleware enforces `isAdminOrUp` (`ToolsController.js:160-163`); the
controller-side validations are at `:30-48`. Background work runs in
`AbMergeManager` (`server/managers/AbMergeManager.js`), which calls
`TaskManager.addTask` on start and `TaskManager.taskFinished` on
completion — *removing the task from the in-memory list both on
success and on failure* (`TaskManager.js:33-37`,
`AbMergeManager.js:298`). The task disappears either way; the
in-progress version is the only one observable via `/api/tasks`.

## Command surface

```
items encode-m4b start  --id <itemId> [--codec copy|aac|opus] [--bitrate 32k|64k|128k|192k] [--channels 1|2]
items encode-m4b cancel --id <itemId>
```

- Exactly one of the two leaf verbs is invoked per call. `--id` is
  required on both.
- `--codec` / `--bitrate` / `--channels` on `start` are optional. CLI
  sends only what the caller explicitly set; ABS applies its own
  defaults (`aac` / `128k` / `2` — `ffmpegHelpers.js:443-445`) for the
  rest.
- Each option is a closed enum at the CLI layer; unknown values are
  rejected before any HTTP call. The web UI's "advanced" view that
  exposes free-form ffmpeg strings is **not** mirrored — agents who
  need it can curl directly. If a real workflow demands it later, we
  can widen.

### `--codec copy` semantics

Remux without re-encode. Source audio must already be AAC for the
result to be a valid `.m4b`. If the source isn't AAC the server-side
ffmpeg pass will fail and the task disappears with an error event the
CLI cannot observe. Briefly noted in `--help`.

## Output

### `start` — happy path

CLI-synthesized receipt to stdout, exit 0. No round-trip to
`/api/tasks` on the default path.

```json
{
  "libraryItemId": "li_x",
  "action": "encode-m4b",
  "started": true,
  "options": { "codec": "aac", "bitrate": "128k", "channels": 2 }
}
```

`options` echoes only what the caller passed. Omitted CLI flags are
omitted from the receipt — readers can rely on "absent = server
default applies".

The receipt is **not** a guarantee that the task has been registered
server-side. ABS calls `AbMergeManager.startAudiobookMerge` without
`await` and then `res.sendStatus(200)` (`ToolsController.js:52-54`).
The receipt only confirms ABS accepted the request.

### `cancel` — happy path

Empty stdout, exit 0. Mirrors ABS's empty `200`.

### Failure paths

See "Error handling" below.

## Observability (what the CLI does *not* do)

ABS's task lifecycle makes the obvious "wait for completion" pattern
brittle:

- `GET /api/tasks` has no per-task GET, no filter by `action` or
  `libraryItemId`, no filter by status. `?include=queue` is the only
  parameter and is unrelated to encode-m4b.
- Tasks are removed from the list on completion **including on
  failure** (`TaskManager.taskFinished` is called by
  `AbMergeManager.removeTask` after every `setFailed`/`setFinished`).
  There is no lingering record to inspect after the fact.
- The ABS web UI uses a WebSocket (`task_finished` event with the
  final `task.toJSON()` including `isFailed` + `error`) to observe
  this. The CLI is HTTP-only; subscribing to that socket is out of
  scope.

Consequences the CLI does not paper over:

1. **No `--wait` flag.** Any polling implementation has a race
   (the task can transition from running to gone between two polls)
   and on failure has no error message to surface. Callers who want to
   wait can `abs-cli tasks list | jq` in a loop themselves. The agent
   keeps the policy; the CLI keeps the primitive.
2. **No after-the-fact "did task X succeed" lookup.** Callers must
   inspect the library item (`abs-cli items get --id <id>`): if
   `media.audioFiles` is a single `.m4b`, the encode succeeded; if it
   still shows the original tracks, the encode either hasn't run yet,
   is still running, or failed silently.
3. **No CLI-side concurrency throttle.** ABS does **not** queue
   encode-m4b tasks across items (`ToolsController.js:45-48` only
   refuses if the **same** item already has a pending task). Firing
   `start` against ten items will spawn ten parallel ffmpeg processes
   on the ABS host. Callers wanting to batch must serialize
   themselves. This is a server reality, surfaced in `--help`.

## Error handling

The CLI relies on `AbsApiClient.cs`'s existing status-code mapping
(`AbsApiClient.cs:280-290`). New behaviour is limited to one parameter
addition on the client and one well-placed `permissionHint` /
`notFoundHint`.

| Status | Cause | CLI behaviour | Exit |
|--------|-------|---------------|------|
| 200 | Success | Receipt (start) / empty stdout (cancel) | 0 |
| 400 "Audiobook is already processing" | Same item already has a pending encode-m4b task (`ToolsController.js:45-48`) | stderr `"Bad request. Audiobook is already processing"` (generic 400 passthrough) | 2 |
| 400 "Invalid library item: not a book" | Podcast / other media type (`:35-38`) | stderr passthrough | 2 |
| 400 "Invalid audiobook: no audio tracks" | Item is a book but has no audio (`:40-43`) | stderr passthrough | 2 |
| 403 | Caller is not admin (`ToolsController.middleware`) | stderr `"Permission denied. This operation requires admin permission."` via `permissionHint` | 2 |
| 404 (start) | Item missing / invalid (`:30-33`) | stderr `"Not found. Audiobook not found"` (generic 404 passthrough) | 2 |
| 404 (cancel) | No pending task **or** item missing (both branches return plain `res.sendStatus(404)` with no body) | stderr `"Not found. no pending encode-m4b task for item <id>, or item does not exist"` via new `notFoundHint` | 2 |

CLI-side validation (before any HTTP call):
- `--id` required.
- `--codec` / `--bitrate` / `--channels` against the enum sets above;
  unknown values rejected with exit 1 and a stderr message naming the
  bad value and the allowed set.

## Components

### New: `src/AbsCli/Services/EncodeM4bService.cs`

Static-instance service following the project's per-domain pattern.
Methods:

```csharp
public Task<EncodeM4bStartReceipt> StartAsync(string itemId, EncodeM4bOptions options);
public Task CancelAsync(string itemId);
```

`StartAsync` POSTs to the endpoint with a query string built only from
options the caller explicitly set, then returns a receipt synthesised
from the request inputs (no follow-up HTTP call). `CancelAsync`
DELETEs and lets `AbsApiClient` surface the response or error; on 404
it passes a `notFoundHint` so the generic handler can produce the
combined message.

### New: `src/AbsCli/Models/EncodeM4bModels.cs`

Two model types:

- `EncodeM4bStartReceipt` — `{ LibraryItemId, Action, Started,
  Options: EncodeM4bOptions }`. `Action` is the string `"encode-m4b"`;
  `Started` is `true` whenever the receipt is constructed (the value
  exists for `jq` discoverability — a future failure-receipt could
  reuse the same shape with `started: false` rather than throwing).
- `EncodeM4bOptions` — `{ Codec?, Bitrate?, Channels? }` (all
  nullable; `null` means "not passed by caller").

Both registered in `AppJsonContext` (`Models/JsonContext.cs`) so AOT
serialisation works without reflection. Self-test
(`SelfTestCommand.cs`) adds a `=== Encode M4B Models ===` round-trip
group.

### New: `ItemsCommand.CreateEncodeM4bCommand()`

Method added to `src/AbsCli/Commands/ItemsCommand.cs`. Returns a
`Command("encode-m4b", "...")` with two subcommands (`start`,
`cancel`) wired in. Registered as a subcommand of `items` alongside
the existing `list`, `get`, `update`, `cover`, etc.

Per project convention: each subcommand has its own examples
(`AddExamples`) and, where applicable, a response-shape sample
(`AddResponseExample<EncodeM4bStartReceipt>` on `start`). Cancel has
no response example — it returns empty.

### Modified: `src/AbsCli/Api/ApiEndpoints.cs`

One constant added:

```csharp
public const string ToolsItemEncodeM4b = "/api/tools/item/{0}/encode-m4b"; // POST start, DELETE cancel
```

### Modified: `src/AbsCli/Api/AbsApiClient.cs`

One parameter added to the existing failure handler so a calling
service can attach a per-call message that gets composed with the
generic `"Not found."` prefix — mirroring the existing
`permissionHint` pattern:

```csharp
// in the response-failure switch (around line 280):
404 when notFoundHint != null => $"Not found. {notFoundHint}",
404 => $"Not found.{(string.IsNullOrWhiteSpace(body) ? "" : $" {body.Trim()}")}",
```

The parameter is opt-in; existing call sites that don't pass
`notFoundHint` behave identically to today.

## AOT considerations

Project ships as Native AOT (`PublishAot=true`). The discipline this
spec inherits:

- **No reflection-based JSON.** Both new models registered in
  `AppJsonContext`; all serialisation through
  `AppJsonContext.Default.<TypeName>`.
- **No `System.Text.Json` reflection overloads.** The receipt is
  produced via the source-generated serialiser.
- **No System.CommandLine handler reflection.** Both leaf commands use
  the existing `SetAction(async (parseResult, ct) => …)` style already
  in the codebase.

Nothing new about this work is AOT-risky; same lane as `items cover`.

## `--help` text constraint

Calibrate against the leaner existing commands (e.g. `items scan`,
`items cover get`). Terse one-liners, no prose blocks, no
"useful when…" framing. Caveats from above surface as the shortest
accurate sentence — not a discussion. Caveats that must appear in
`--help` somewhere on the verb tree:

- Admin-only (both leaf verbs).
- Fire-and-forget — observability via `tasks list` (only while
  running) and `items get` (after completion).
- No concurrency guardrail across items — caller serialises for
  batch.
- `--codec copy` requires AAC source.
- `cancel` 404 = no task pending or item doesn't exist (combined
  message — the API can't distinguish).
- Originals moved (not deleted) to ABS's metadata cache as backup.

Wording calibration happens at implementation time, not in this spec.

## Testing

### Unit tests (`tests/AbsCli.Tests`)

- Enum validation per option on `start`: each known good value
  accepted; arbitrary string rejected with the right exit code before
  any HTTP call.
- Receipt shape: `options` echoes only what the caller passed (omitted
  flags absent in JSON).
- Query-string assembly: only set options appear in the URL.
- `--id` required on both `start` and `cancel`.
- Self-test (`SelfTestCommand`) round-trip group for the two new
  models.

### Smoke test (`docker/smoke-test.sh`)

The encode requires real audio frames, not the synthetic 38-frame
silent MP3 that `seed.sh` produces for upload tests. Fixtures are
generated inside the smoke (not seeded), so the test owns its setup
and teardown end to end.

**Fixture generation** (ffmpeg available in both CI's
`ubuntu-latest` runner and the dev container — added to
`.devcontainer/Dockerfile` as part of this work):

```bash
ffmpeg -y -f lavfi -i "sine=frequency=440:duration=3" -c:a libmp3lame -b:a 128k track1.mp3
ffmpeg -y -f lavfi -i "sine=frequency=523:duration=3" -c:a libmp3lame -b:a 128k track2.mp3
```

Two ~50KB MP3s with real frames; AAC re-encode by ABS's ffmpeg works
on these (verified locally before merge).

**Flow:**

1. Generate `track1.mp3` + `track2.mp3` to a `mktemp -d` tempdir
   using the two `ffmpeg -f lavfi -i sine=...` invocations above.
   Install `trap` for cleanup so the tempdir + ABS item are removed
   on any exit path (matches the existing patterns at
   smoke-test.sh:539-720).
2. `abs-cli upload --title "ENCODE_M4B_TEST" --wait --files track1.mp3 track2.mp3`
   → capture library item id from the `LibraryItemMinified` response.
3. Assert `abs-cli items get --id $ID` shows
   `media.audioFiles.length == 2`.
4. `abs-cli items encode-m4b start --id $ID --codec copy` → assert
   receipt shape (`started: true`, `options.codec == "copy"`,
   `libraryItemId == $ID`).
5. Poll `abs-cli tasks list` until no task with
   `action == "encode-m4b"` and `data.libraryItemId == $ID` remains.
   Timeout ~60s; failure of the timeout fails the smoke.
6. `abs-cli items get --id $ID` → assert
   `media.audioFiles.length == 1` and the single file's
   `metadata.filename` ends in `.m4b`.
7. **Cleanup** (mirrors smoke-test.sh:568/594/667/689):
   `curl -sf -X DELETE "$ABS_URL/api/items/$ID?hard=1" -H "$AUTH"`
   then `rm -rf` the tempdir. Wrapped in the `trap` from step 1.

**Negative paths** (same flow extended):

- Already-processing 400: second `start` immediately after the first
  (before the task finishes / before cleanup) exits 2 with stderr
  containing `"Audiobook is already processing"`.
- Cancel 404: `abs-cli items encode-m4b cancel --id <nonexistent>`
  exits 2 with stderr containing
  `"no pending encode-m4b task for item"`.

**Permission denial gate** (uses the existing `uploaduser` non-admin
token already wired into the smoke):
- `abs-cli items encode-m4b start --id $ID` as `uploaduser` exits 2
  with the admin permission-denied message.
- No cleanup needed (the call failed; nothing was created).

## Out of scope (deferred)

These were considered and explicitly rejected for v0.5.0. Each can be
picked up in a future spec if a real workflow demands it.

- **`--wait` and any task polling baked into the CLI.** Server-side
  observability is too lossy for the CLI to wrap reliably; agents
  poll themselves if they want to wait.
- **Free-form `--codec` / `--bitrate` / `--channels` values
  (advanced view).** Enum coverage matches the ABS UI presets; widen
  on demand.
- **Batch / multi-item start.** ABS has no batch endpoint and no
  concurrency throttle; a CLI-side fan-out would have to serialise,
  which is what an agent loop already does.
- **`items delete` for cleanup.** Smoke uses raw `curl -X DELETE`,
  matching the established pattern in `smoke-test.sh`. A real
  `items delete` command needs its own design (hard vs. soft delete,
  cascade semantics).
- **WebSocket subscription for progress.** Out of scope for the
  HTTP-only CLI.

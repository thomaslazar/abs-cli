# `collections` — design

Status: drafted 2026-05-22. Targets v0.6.0.

## Summary

Add a top-level `collections` command with ten verbs that wrap ABS's full
collections endpoint set:

```
collections list   [--library <id>] [--limit] [--page] [--include rssfeed]
collections get    --id <cid> [--include rssfeed]
collections create [--library <id>] --name <n> [--description <d>] (--input <file>|--stdin)
collections update --id <cid> [--name <n>] [--description <d>|""]
collections reorder --id <cid> (--input <file>|--stdin)
collections delete --id <cid>
collections add    --id <cid> --book <libraryItemId>
collections remove --id <cid> --book <libraryItemId>
collections batch-add    --id <cid> (--input <file>|--stdin)
collections batch-remove --id <cid> (--input <file>|--stdin)
```

Collections are flat, manually-curated, library-scoped ordered lists of
book library items (see research:
[docs/specs/research/2026-05-22-collections.md](research/2026-05-22-collections.md)).
ABS has no smart-collection / saved-filter concept — membership is the
agent's job. The CLI ships ten primitives that map 1:1 onto ABS endpoints,
with two deliberate exceptions documented in the research doc:

1. The global `GET /api/collections` is **dropped** in favour of the
   library-scoped paginated endpoint (consistency with every other
   `list` verb).
2. ABS's `PATCH /api/collections/:id` is **split into two CLI verbs**
   (`update` for name/description, `reorder` for the books array) because
   it overloads two unrelated mental models behind one URL.

All membership-change endpoints (single + batch, add + remove) remain
separate verbs — folding them would hide divergent error semantics
(single-add 400s on duplicate; batch-add silently skips).

## Motivation

Collections are an established ABS feature, exposed in the web UI under
each library's "Collections" tab. The CLI does not currently expose them
at all — they sit in `docs/roadmap.md`'s "Deferred features" table with
the reason "not needed for metadata workflow." The agent harness around
the CLI now wants curated lists (e.g. genre-driven sets, themed
groupings) to drive downstream workflows. Building those lists requires
all four verb families: read (list/get), create, update, membership
edit. Anything less leaves the agent half-blind to its own data.

## ABS endpoints touched

Nine ABS endpoints (eight in `CollectionController.js`, one in
`LibraryController.js`); the CLI exposes ten verbs because `update` and
`reorder` share `PATCH /api/collections/:id`. The ninth
`CollectionController` endpoint (`GET /api/collections`, global) is
deliberately not used — see Summary.

| CLI verb | Endpoint | Body | Permission |
|---|---|---|---|
| `list` | `GET /api/libraries/:id/collections` | – | (read) |
| `get` | `GET /api/collections/:id` | – | (read) |
| `create` | `POST /api/collections` | `{libraryId,name,description?,books:[...]}` | `update` |
| `update` | `PATCH /api/collections/:id` | `{name?,description?}` | `update` |
| `reorder` | `PATCH /api/collections/:id` | `{books:[...]}` | `update` |
| `delete` | `DELETE /api/collections/:id` | – | `delete` |
| `add` | `POST /api/collections/:id/book` | `{id:libraryItemId}` | `update` |
| `remove` | `DELETE /api/collections/:id/book/:libraryItemId` | – | `update` |
| `batch-add` | `POST /api/collections/:id/batch/add` | `{books:[...]}` | `update` |
| `batch-remove` | `POST /api/collections/:id/batch/remove` | `{books:[...]}` | `update` |

Routing: `temp/audiobookshelf/server/routers/ApiRouter.js:80` and
`:146-154`.

### Server-side dead query params on `list`

`getCollectionsForLibrary` (`LibraryController.js:814-844`) accepts
`sort`, `desc`, `filter`, and `minified` query params, but **none of them
affect the result**. They are stored in the response payload object as
echo-back (lines 825-828), but the model call
`getOldCollectionsJsonExpanded` ignores them entirely. The TODO comment
on line 832 (`// TODO: Create paginated queries`) confirms this is
stubbed-but-unimplemented upstream. Only `limit`, `page`, and
`include=rssfeed` are functional.

The CLI surface therefore exposes only `--limit`, `--page`, and
`--include rssfeed` on `list`. Sort/desc/filter/minified come back in a
future release when ABS implements them.

## Command surface

### `list`

```
collections list [--library <id>] [--limit <n>] [--page <n>] [--include rssfeed]
```

| Flag | Required | Default | Notes |
|---|---|---|---|
| `--library` | no | from config | Standard library override — same pattern as `series list`, `authors list`. |
| `--limit` | no | 50 | Forwarded to ABS. Post-fetch slice; not a true SQL limit. |
| `--page` | no | 0 | 0-indexed. |
| `--include` | no | – | Value `rssfeed` decorates each row with the collection's open RSS feed (if any). |

### `get`

```
collections get --id <cid> [--include rssfeed]
```

`--id` required. `--include rssfeed` decorates with RSS feed state.

### `create`

```
collections create --library <id> --name <n> [--description <d>] (--input <file>|--stdin)
```

| Flag | Required | Notes |
|---|---|---|
| `--library` | no | From config; standard override. |
| `--name` | yes | Server strips HTML (`htmlSanitizer.stripAllTags`). |
| `--description` | no | Free text; empty string allowed at create. |
| `--input` | yes* | JSON file path: `{"books":["lid",…]}`. |
| `--stdin` | yes* | Same JSON shape from stdin. |

*Exactly one of `--input` or `--stdin` is required; ABS rejects empty
books arrays with 400 ("Invalid collection data. No books").

### `update`

```
collections update --id <cid> [--name <n>] [--description <d>|""]
```

| Flag | Required | Notes |
|---|---|---|
| `--id` | yes | Collection ID. |
| `--name` | no | Empty string is rejected client-side. |
| `--description` | no | Tri-state: omit = unchanged, `""` = clear (sent as JSON `null`), non-empty = set. |

At least one of `--name` / `--description` is required (matches
`authors update`'s ergonomic rule).

### `reorder`

```
collections reorder --id <cid> (--input <file>|--stdin)
```

Input body: `{"books":["lid",…]}` — full membership in desired order.
Partial lists shuffle missing members to undefined positions; see the
`--help` text constraint section below.

### `delete`

```
collections delete --id <cid>
```

`--id` required. Prints `{"success":"true"}` on 200 (matching
`authors delete` — `AuthorsCommand.cs:245`).

### `add` / `remove`

```
collections add    --id <cid> --book <libraryItemId>
collections remove --id <cid> --book <libraryItemId>
```

Both flags required on both verbs. `--book` is a libraryItemId.

### `batch-add` / `batch-remove`

```
collections batch-add    --id <cid> (--input <file>|--stdin)
collections batch-remove --id <cid> (--input <file>|--stdin)
```

Input body: `{"books":["lid",…]}` (verbatim ABS shape). Matches
`items batch-update --input/--stdin` precedent.

## Components

### New: `src/AbsCli/Commands/CollectionsCommand.cs`

Standard System.CommandLine command class modeled on
`SeriesCommand.cs` / `AuthorsCommand.cs`. Ten subcommands. Each:

- Wires options.
- Calls `CommandHelper.BuildClient(libraryOverride: …)` (for `list` and
  `create`; the rest don't take a library context).
- Calls into `CollectionsService`.
- Pipes the response through `ConsoleOutput.WriteJson(result, JsonContext…)`.
- Tags non-read verbs via `command.AddPermissionRequired(...)` per
  `CLAUDE.md` → "Permission tagging".
- Adds `--help` blocks per `feedback_help_documents_caveats` (see
  "`--help` text constraint" below).

### New: `src/AbsCli/Services/CollectionsService.cs`

Modeled on `AuthorsService.cs`. Ten methods, one per endpoint. Pattern:

```csharp
public async Task<Collection> CreateAsync(string libraryId, string name, string? description, string booksJson)
public async Task<PaginatedResponse> ListAsync(string libraryId, int limit, int? page, string? include)
public async Task<Collection> GetAsync(string id, string? include)
public async Task<Collection> UpdateAsync(string id, Dictionary<string, string> body)
public async Task<Collection> ReorderAsync(string id, string booksJson)
public async Task DeleteAsync(string id)
public async Task<Collection> AddBookAsync(string id, string libraryItemId)
public async Task<Collection> RemoveBookAsync(string id, string libraryItemId)
public async Task<Collection> BatchAddAsync(string id, string booksJson)
public async Task<Collection> BatchRemoveAsync(string id, string booksJson)
```

Permission-hint strings on PATCH / POST / DELETE calls (per
`CLAUDE.md` → "Permission hint mirroring"):
- `'update' permission` for create, update, reorder, add, remove,
  batch-add, batch-remove.
- `'delete' permission` for delete.

The two batch endpoints and `reorder` accept a raw JSON string from the
caller — same passthrough pattern as `items batch-update`. The service
validates that the input is well-formed JSON before sending, but does
not enforce the `{"books":[…]}` shape — that's ABS's job.

```csharp

```

No `RssFeed` DTO exists today (`grep -ri 'RssFeed' src/AbsCli/Models/`
returns nothing). Add a minimal `RssFeed.cs` matching the shape produced
by `Feed.toOldJSON()` server-side — fields ABS populates: `id`, `slug`,
`userId`, `entityType`, `entityId`, `entityUpdatedAt`, `coverPath`,
`serverAddress`, `feedUrl`, `meta` (object), plus minor URL fields. The
CLI only round-trips the shape — no field is acted on — so the DTO can
be tolerant (nullable everywhere, `JsonExtensionData` if the shape
drifts).

## AOT considerations

All ten verbs use the existing source-generated JSON context (`AppJsonContext`).
No reflection, no `JsonSerializer.Deserialize<T>` without a context. The
batch endpoints (`reorder`, `batch-add`, `batch-remove`, and the books
half of `create`) pass through a JSON string the caller provided, which
sidesteps DTO modeling entirely — the service just verifies it parses
and sends.

The PATCH body for `update` uses the same `Dictionary<string, string>`
tri-state pattern as `AuthorsCommand.BuildUpdateBodyForTesting`
(`AuthorsCommand.cs:214-224`). That dictionary serializer is already
registered in `JsonContext`.

## Output formats

Every successful verb prints a single JSON document on stdout:

| Verb | Output |
|---|---|
| `list` | `PaginatedResponse` over `Collection`, expanded shape. |
| `get` | One `Collection`, expanded. |
| `create` | The created `Collection`, expanded. |
| `update` | The updated `Collection`, expanded. |
| `reorder` | The reordered `Collection`, expanded. |
| `delete` | `{"success":"true"}` synthesized on HTTP 200 (ABS returns empty body). |
| `add` / `remove` | The updated `Collection`, expanded. |
| `batch-add` / `batch-remove` | The updated `Collection`, expanded. |

All but `delete` echo what ABS returned verbatim. stderr is reserved for
errors and warnings (existing log format).

## Error handling

Standard CLI error model, no new error types:

| Failure | Exit | stderr |
|---|---|---|
| Auth missing / expired | 1 | Existing `EnsureValidTokenAsync` flow. |
| 401 / 403 | 2 | `Permission denied. This operation requires '<hint>'.` |
| 404 (collection / library item not found) | 2 | `Not found. <server message>` |
| 400 (create with empty books, cross-library add, duplicate single-add) | 2 | `Bad request. <server message>` |
| 500 | 3 | `Server error. <server message>` |
| Malformed `--input` JSON | 1 | `Invalid JSON input: <parse error>` |
| Both `--input` and `--stdin` provided | 1 | `Provide --input or --stdin, not both.` |
| Neither provided where required | 1 | `Provide --input <file> or --stdin.` |
| Empty `--name` on `update` | 1 | `--name cannot be empty.` |
| No editable flags on `update` | 1 | `Specify at least one of --name, --description.` |

The existing `AbsApiClient` already maps server responses to these
categories — no per-command branching needed beyond the input-validation
errors at the top of each subcommand.

## `--help` text constraint

Per `feedback_help_documents_caveats` and `feedback_help_brevity`, each
verb's `--help` must call out the relevant sharp edge in tight prose
(not pasted from this spec). The non-negotiable items:

- `create --help`: "ABS requires at least one book; cannot create empty.
  HTML in --name is stripped silently."
- `update --help`: "Edits metadata only. Use `reorder` to change book
  order; use `add`/`remove`/`batch-*` to change membership."
- `reorder --help`: "Reorders existing members only — does not add or
  remove. Pass the **full** current membership in the desired order;
  partial lists shuffle missing members to undefined positions. Example:
  ..."
- `add --help`: "Errors with 400 if the book is already in the
  collection. Use `batch-add` (or check first) for idempotent inserts."
- `batch-add --help`: "Silently skips books already in the collection,
  unlike `add` which 400s. Books from a different library are rejected."
- `update --help`: "Empty string for --description clears the field;
  omit to leave unchanged. Empty --name is rejected. Same convention as
  `authors update`."
- `list --help`: "ABS echoes `sortBy` / `sortDesc` / `filterBy` /
  `minified` back in the response payload, but these reflect no applied
  behavior today (server-side TODO). Treat them as inert."

Per `feedback_help_self_contained`, no help text enumerates the
`Collection` response shape — `--help` describes the verb's behavior;
response shape lives in the `Response shape:` block driven by
`command.AddResponseExample<…>()`.

### Smoke test additions (`docker/smoke-test.sh`)

Add a `collections` block exercising the golden path end-to-end:

1. `collections list --library <lid>` — empty result OK on fresh seed.
2. `collections create --library <lid> --name "smoke test" --input -` with
   2 seeded library item IDs piped in. Capture the returned `id`.
3. `collections get --id <cid>` — confirm 2 books and the name.
4. `collections update --id <cid> --name "renamed" --description "desc"`
   — confirm fields.
5. `collections add --id <cid> --book <lid3>` — confirm 3 books.
6. `collections add --id <cid> --book <lid3>` — expect 400 (already in
   collection) → CLI exits 2.
7. `collections reorder --id <cid> --input -` piping `{"books":[lid3,lid2,lid1]}`
   — confirm new order.
8. `collections batch-remove --id <cid> --input -` piping
   `{"books":[lid1,lid2,lid3]}` — confirm empty `books`.
9. `collections delete --id <cid>` — expect `{"success":"true"}`.
10. `collections get --id <cid>` — expect 404 → CLI exits 2.

Plus a permission-denied check: as `uploaduser` (no `update`), run
`collections create` and expect exit 2 with the standard `'update'
permission` message. Per `feedback_extend_seed_for_403`, the seed
already has `uploaduser` (no `update` perm) — no seed changes required.

## Files NOT modified

- `docker/seed.sh` — existing seed has enough book library items to
  exercise the smoke flow; no new fixtures needed.
- Other resource models (`Series.cs`, `Author.cs`, etc.) — no
  cross-resource impact.

## Out of scope (deferred)

- **RSS feed endpoints.** `POST /api/feeds/collection/:id/open` and the
  feed-close path are their own resource family. Future feeds-focused
  milestone.
- **Smart / saved-filter collections.** Not an ABS feature; would
  require client-side state (saved query) that violates the
  thin-passthrough rule. Agents handle their own resync.
- **`--from-stdin` for the items-list piping recipe.** The light-novels
  example in the research doc uses `xargs` to convert a list of IDs into
  repeated `--book` flags. With the locked `--input`/`--stdin` shape,
  the recipe is even simpler: `items list … | jq '{books: [.results[].id]}' | collections create … --stdin`.
  No special flag needed.
- **Sort / filter / minified on `list`.** Server-side dead code today.
  Add when ABS implements the TODO at `LibraryController.js:832`.

## Open questions

None outstanding. Resolved during review:

- `update --description ""` clears (silent, mirrors `authors update`);
  documented in `update --help`.
- Inert pagination fields (`sortBy` / `sortDesc` / `filterBy` /
  `minified`) pass through the response verbatim; `list --help` calls

- `RssFeed` DTO does not exist yet — `RssFeed.cs` will be added per the

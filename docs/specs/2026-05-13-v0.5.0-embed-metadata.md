# `items embed-metadata` and `items batch-embed-metadata` — design

Status: approved 2026-05-13. Targets v0.5.0.

## Summary

Add two leaf commands that wrap ABS's "embed metadata into audio
files" tools — one for a single item, one for a batch:

```
abs-cli items embed-metadata       --id <id>                        [--no-backup] [--force-embed-chapters] [--wait]
abs-cli items batch-embed-metadata {--input <file> | --stdin}        [--no-backup] [--force-embed-chapters] [--wait]
```

Both endpoints are admin-only. Both run an in-place ffmpeg rewrite of
the item's audio files to bake the current ABS state (tags, cover,
chapters) into the file metadata. Without these commands, the existing
metadata-write commands (`items update`, `items chapters set`,
`authors update`, etc.) only persist to ABS's DB and sidecar — the
audio files themselves stay untouched.

Third of four v0.5.0 "file management" primitives.

## Motivation

The CLI's existing write commands are explicitly DB-and-sidecar-only.
That's a deliberate ABS design (cheap, transactional, reversible),
but it means anything that reads the audio file outside of ABS — a
different player, a future re-import, a script using `ffprobe` — sees
stale metadata. The Tools → Embed Metadata action in the ABS web UI
bridges this gap; this spec exposes the same primitives to the CLI so
agents can finish a metadata-cleanup loop by flushing ABS state into
the files.

## Command surface

```
items embed-metadata       --id <itemId> [--no-backup] [--force-embed-chapters] [--wait]
items batch-embed-metadata {--input <file> | --stdin} [--no-backup] [--force-embed-chapters] [--wait]
```

### Common flags

- `--no-backup` (default off → backup applied) — opts out of the
  server-side pre-rewrite backup. The endpoint takes `backup=1` to
  copy each audio file into `<MetadataPath>/cache/items/<id>/<filename>`
  before the in-place rewrite. ABS's web UI defaults this on; the CLI
  matches. Pairs with v0.5.0's planned cache purge primitive for
  cleanup of the accumulated cache.
- `--force-embed-chapters` (default off) — opt-in to embed chapters
  for multi-file items. ABS's `AudioMetadataManager` only auto-includes
  chapters when `audioFiles.length == 1`; multi-file items get tags +
  cover only unless this flag is set. With the flag on, every track
  ends up carrying the full chapter list — every player picking any
  track sees the same navigation. That's a real content choice, not a
  passthrough quirk, hence opt-in.
- `--wait` — block until the matching `embed-metadata` task(s)
  disappear from `/api/tasks`. See "Wait semantics" below.

### `embed-metadata` (single)

- `--id` required.

### `batch-embed-metadata`

- Exactly one of `--input <file>` and `--stdin` required. Mirrors the
  established batch shape (`items batch-update` / `items batch-get`
  / `items chapters set`).
- Input is the ABS request body verbatim:
  `{ "libraryItemIds": ["li_a", "li_b", ...] }`.
- Validation: AOT JSON deserialization through a typed
  `BatchEmbedMetadataRequest` model is the shape gate. Empty array
  or wrong types fail at deserialization (exit 1, before HTTP).

## Wait semantics

When `--wait` is passed, the CLI polls `GET /api/tasks` every 2s
until **all** tasks matching the invocation have disappeared. The
match predicate is:

```
task.action == "embed-metadata" && data.libraryItemId IN <ids>
```

For `embed-metadata`, `<ids>` is a single-element set with `--id`'s
value. For `batch-embed-metadata`, it's the full input list.

**Timeout: 600s (10 min)** hard-coded. If the polling window exceeds
the timeout with tasks still pending, the CLI exits 2 with stderr
"Timed out waiting for embed-metadata task(s) to complete". The
timeout is generous enough for a multi-hour multi-track audiobook on
slow disks; callers wanting different behavior can fire without
`--wait` and poll themselves.

**Critical caveat (documented in `--help`):** ABS removes the task
from `/api/tasks` on completion regardless of success or failure
(`TaskManager.taskFinished` is called by `AudioMetadataManager` after
both `setFinished` and `setFailed`). So task-disappeared means **ABS
stopped processing it**, not that the embed succeeded. The CLI cannot
distinguish — there is no post-task GET, no per-task success flag,
and the WebSocket events that the web UI consumes are out of scope
for the HTTP-only CLI. `--wait` exiting 0 means processing ended;
verification of the actual on-disk file requires `ffprobe` or
similar, which the agent owns.

## Output

### `embed-metadata` — happy path

CLI-synthesized receipt to stdout, exit 0. No follow-up HTTP call.

```json
{
  "libraryItemId": "li_x",
  "action": "embed-metadata",
  "started": true,
  "options": { "backup": true, "forceEmbedChapters": false }
}
```

`options` always carries both keys (unlike encode-m4b's sparse echo
of caller-provided flags). The booleans reflect the effective values
the CLI sent on the query string. With `--wait`, the receipt is
printed only after the task disappears from `/api/tasks` (so its
presence implies "ABS stopped processing"); without `--wait` it
prints immediately after the POST returns.

### `batch-embed-metadata` — happy path

Similar receipt with the full input list:

```json
{
  "action": "embed-metadata",
  "started": true,
  "libraryItemIds": ["li_a", "li_b"],
  "options": { "backup": true, "forceEmbedChapters": false }
}
```

### Failure paths

See "Error handling" below.

## Error handling

The CLI relies on `AbsApiClient.cs`'s existing status-code mapping; no
new client-side changes required.

### `embed-metadata` (single)

| Status | Cause | CLI behaviour | Exit |
|--------|-------|---------------|------|
| 200 | Success | Receipt to stdout | 0 |
| 400 | Item missing / not a book / no audio tracks (`ToolsController.js:85-88`) | Generic 400 passthrough | 2 |
| 400 "Library item is already in queue or processing" | Same item already queued or running (`ToolsController.js:90-93`) | Generic 400 passthrough (ABS includes the message in the body) | 2 |
| 403 | Caller is not admin (`ToolsController.middleware`) | `Permission denied. This operation requires admin permission.` via `permissionHint` | 2 |
| 404 | Item ID not found (`ToolsController.middleware:166-170`) | Generic 404 passthrough | 2 |

### `batch-embed-metadata`

| Status | Cause | CLI behaviour | Exit |
|--------|-------|---------------|------|
| 200 | Success | Receipt to stdout | 0 |
| 400 "Invalid request payload" | Empty `libraryItemIds` (`ToolsController.js:114-116`) | Generic 400 passthrough | 2 |
| 400 (per-item invalid) | Any one item missing/not-book/no-audio (`:132-135`) | Generic 400 passthrough — message identifies the offending item ID | 2 |
| 400 "Library item is already in queue or processing" | Any one item already queued/running (`:137-140`) | Generic 400 passthrough | 2 |
| 403 | Caller is not admin **or** lacks access to any one item (`:127-130`) | `Permission denied. This operation requires admin permission.` via `permissionHint` | 2 |
| 404 | Any one item ID not found (`:121-124`) | Generic 404 passthrough | 2 |

The batch endpoint validates all items upfront, so a single bad ID
aborts the whole batch with no partial-success state — that's an
ABS-side semantic, which the CLI surfaces unchanged.

### `--wait` failure modes

| Cause | CLI behaviour | Exit |
|---|---|---|
| Polling exceeds 600s timeout | stderr `"Timed out waiting for embed-metadata task(s) to complete"` | 2 |
| Polling hits HTTP failure on `/api/tasks` | Generic `AbsApiClient` error surfacing | 2 |

CLI-side validation (before any HTTP call):
- `embed-metadata`: `--id` required.
- `batch-embed-metadata`: exactly one of `--input` / `--stdin`;
  request deserializes into `BatchEmbedMetadataRequest`; both checks
  reject with exit 1 before HTTP.

## Components

### New: `src/AbsCli/Services/EmbedMetadataService.cs`

Static-instance service following the per-domain pattern. Methods:

```csharp
public Task<EmbedMetadataReceipt>      StartAsync(string itemId, EmbedMetadataOptions options);
public Task<BatchEmbedMetadataReceipt> StartBatchAsync(BatchEmbedMetadataRequest request, EmbedMetadataOptions options);
public Task WaitForCompletionAsync(IReadOnlyList<string> libraryItemIds, TimeSpan timeout, CancellationToken ct);
```

`StartAsync` POSTs to the single-item endpoint with a query string
built from the options, then returns a CLI-synthesized receipt.
`StartBatchAsync` POSTs the JSON body to the batch endpoint with the
same query-string options. Both use `AbsApiClient.PostAsync` /
`PostEmptyAsync` with `permissionHint: "admin permission"`.

`WaitForCompletionAsync` calls `TasksService.ListAsync()` in a loop,
filtering for `action == "embed-metadata"` and
`data.libraryItemId IN ids`. Returns when zero matching tasks remain
or throws `TimeoutException` after the timeout. Polling interval:
2s. The command-level handler wraps the timeout into the stderr
message and exit 2.

### New: `src/AbsCli/Models/EmbedMetadataModels.cs`

Four model types:

- `EmbedMetadataOptions` — `{ Backup, ForceEmbedChapters }`. Both
  non-nullable bools; CLI always sends both query params explicitly
  (`backup={0|1}&forceEmbedChapters={0|1}`) for unambiguous behavior.
- `EmbedMetadataReceipt` — `{ LibraryItemId, Action="embed-metadata",
  Started=true, Options: EmbedMetadataOptions }`. Single-item receipt.
- `BatchEmbedMetadataRequest` — `{ LibraryItemIds: List<string> }`.
  Deserialization target for `--input`/`--stdin` AND the body posted
  to ABS.
- `BatchEmbedMetadataReceipt` — `{ Action="embed-metadata",
  Started=true, LibraryItemIds: List<string>, Options:
  EmbedMetadataOptions }`. Batch receipt.

All four serializable types registered in `AppJsonContext`
(`Models/JsonContext.cs`). Self-test
(`SelfTestCommand.cs`) gets an `=== Embed Metadata Models ===`
round-trip group.

### New: `ItemsCommand.CreateEmbedMetadataCommand()` and `ItemsCommand.CreateBatchEmbedMetadataCommand()`

Two methods on `src/AbsCli/Commands/ItemsCommand.cs`. Each is a leaf
verb (no nested subcommands), registered under `items` alongside
`encode-m4b`, `chapters`, etc.

Per project convention: each has its own `AddExamples`, a `Caveats`
help section, and `AddResponseExample<T>` pointing at the relevant
receipt type.

### Modified: `src/AbsCli/Api/ApiEndpoints.cs`

Two constants added:

```csharp
public static string ToolsItemEmbedMetadata(string id) => $"/api/tools/item/{id}/embed-metadata";
public const string ToolsBatchEmbedMetadata = "/api/tools/batch/embed-metadata";
```

### Reused: `TasksService` (no changes)

The `--wait` polling reads `GET /api/tasks` through the existing
`TasksService`. If the service doesn't already expose a list method
that returns the typed `TaskListResponse`, we use what's already
there — no changes for this spec.

## AOT considerations

Project ships as Native AOT (`PublishAot=true`). Standard discipline:

- All new models registered in `AppJsonContext`; serialization
  through `AppJsonContext.Default.<TypeName>`.
- No reflection-based JSON overloads.
- No System.CommandLine handler reflection — both commands use the
  established `SetAction(async (parseResult, ct) => …)` style.

Nothing new about this work is AOT-risky; same lane as `items
encode-m4b`.

## `--help` text constraint

Calibrate against the leaner existing commands (`items scan`,
`items chapters set`). Caveats that must surface in `--help`:

**`items embed-metadata`:**
- Admin only.
- In-place destructive rewrite. `--backup` defaults on; only pass
  `--no-backup` deliberately.
- Multi-file items: chapters embedded only with
  `--force-embed-chapters`. Default behavior writes tags + cover per
  track but no chapters.
- `--wait` exits 0 when ABS stops processing — that does **not**
  guarantee success. ABS has no post-task success signal.
- ABS internally queues at `MAX_CONCURRENT_TASKS`; this is fine for
  parallel calls but `--wait` may sit in queue first.

**`items batch-embed-metadata`:**
- Same caveats as above.
- Body must be `{"libraryItemIds":[...]}`.
- Batch validates **all** items upfront — any one bad ID aborts the
  whole batch with 400/403/404 before any processing starts.

## Testing

### Unit tests (`tests/AbsCli.Tests`)

- Model round-trip for each of `EmbedMetadataOptions`,
  `EmbedMetadataReceipt`, `BatchEmbedMetadataRequest`,
  `BatchEmbedMetadataReceipt`.
- `batch-embed-metadata`: malformed JSON (wrong type for
  `libraryItemIds`) rejected with exit 1 before HTTP.
- `batch-embed-metadata`: `--input` and `--stdin` mutual exclusion;
  error when both or neither given.
- Help text: both verbs render the flags, the response shape, and
  the four/five caveats from above.
- Self-test (`SelfTestCommand`) round-trip group for all four
  serializable types.

### Smoke test (`docker/smoke-test.sh`)

Same fixture pattern as `encode-m4b` smoke (two ffmpeg-generated MP3
fixtures uploaded as a multi-file item, plus a single-file item via
the existing seed).

**Single happy path (with `--wait`):**
1. Upload a real-audio fixture, capture item ID.
2. `abs-cli items embed-metadata --id $ID --wait` → receipt printed,
   exit 0.
3. Within ~30s, assert no `embed-metadata` task remains in
   `tasks list` (already implied by --wait having returned).

**Batch happy path:**
1. Upload two real-audio fixtures (or use the encode-m4b cancel-test
   item plus the newly uploaded one).
2. `echo '{"libraryItemIds":["$ID1","$ID2"]}' | abs-cli items batch-embed-metadata --stdin --wait`
   → receipt prints with both IDs, exit 0.

**Negative paths:**
- Single 400 on missing item: `abs-cli items embed-metadata --id li_nonexistent` exits 2 with stderr containing "Bad request" or "Not found" (404 vs 400 depends on middleware order; both are acceptable).
- Single 400 on already-processing: fire two starts back-to-back
  without `--wait`; the second exits 2 with stderr containing
  `"already in queue or processing"`.
- Batch 400 on empty list: `echo '{"libraryItemIds":[]}' | abs-cli items batch-embed-metadata --stdin` exits 2 with `"Invalid request payload"`.
- Batch 404 with one bad ID in the array: whole batch aborts.
- Mutual exclusion: `--input X --stdin` (both) and neither flag both
  error with exit 1 before HTTP.
- `--wait` timeout (synthetic): forced by querying with a very tight
  timeout via an env-var hook OR documented as untested-in-smoke if
  10 min is impractical to provoke in CI. **Plan**: skip in smoke,
  cover semantics in a unit test (mock-free since we just need to
  verify the timeout path triggers).

**Permission denial (admin gate):**
- `abs-cli items embed-metadata --id $ID` as `uploaduser` exits 2
  with `"This operation requires admin permission"` via `permissionHint`.

**Backup behavior:**
- After a successful single embed with default `--backup`, assert the
  ABS metadata cache contains `<MetadataPath>/cache/items/<id>/`
  (via `docker exec` inspection — same pattern the encode-m4b smoke
  uses for cache verification). With `--no-backup`, the cache dir
  for that item should NOT be created. **Note**: this is the only
  smoke assertion that reaches into the container filesystem; if
  brittle, document as "manual verification" instead.

## Out of scope (deferred)

- **WebSocket subscription for true success/failure signaling.**
  HTTP-only CLI; the WebSocket task-finished event with `isFailed`
  is out of scope.
- **CLI-side concurrency throttle for batch.** ABS internally queues
  via `MAX_CONCURRENT_TASKS`; CLI fan-out is fine to leave to the
  agent.
- **`--wait` polling interval flag.** Hard-coded 2s. No workflow has
  demanded tuning yet.
- **`--wait` timeout flag.** Hard-coded 600s. If a real workflow
  needs longer, widen via spec follow-up.
- **`embed-metadata` returning task IDs.** ABS returns empty 200;
  reproducing task IDs would require reading the just-created
  `/api/tasks` entries between start and the first `--wait` poll,
  which is racy. Receipt carries `libraryItemId`(s); that's the
  observable identifier.
- **Verifying ffprobe-level success on the file.** Out of scope —
  agent's responsibility if the workflow requires it.

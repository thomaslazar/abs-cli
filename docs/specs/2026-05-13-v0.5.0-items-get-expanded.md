# `items get --expanded` — design

Status: approved 2026-05-13. Targets v0.5.0.

## Summary

Add an opt-in `--expanded` flag to `abs-cli items get` that switches the
response from ABS's minified shape (the current default) to the
expanded shape. The expanded response carries the `libraryFiles[]`
array — the file-level metadata an agent needs to discover ebook
file inos for `items toggle-ebook-status`, to inspect what files an
item actually has on disk, and so on.

The flag is off by default, so every existing caller's behavior is
unchanged. Other minified-returning commands (`items list`,
`items batch-get`, `search`) are out of scope for this spec — see the
"Scope" section for the rationale.

## Motivation

The CLI today is silent on file-level item state. `items get` returns
the minified shape, which has no `libraryFiles[]`. The expanded
fields are reachable only by hitting ABS directly:

```bash
curl -sf "$ABS_URL/api/items/<id>?expanded=1" -H "$AUTH" | jq .libraryFiles
```

The `items toggle-ebook-status` smoke test from the previous PR
exposes this gap concretely: it can't use the CLI to discover a
supplementary ebook file's `ino`, so it falls back to raw curl with
`?expanded=1` for the initial state read. Any agent driving the
toggle command end-to-end faces the same wall.

This spec closes that hole by surfacing the expanded shape through
the CLI itself.

## Scope

`items get --expanded` only. The other minified-returning commands
are deliberately out of scope:

- **`items list`** — the ABS list endpoint (`GET /api/libraries/:id/items`)
  does not return `libraryFiles[]` regardless of `?expanded=1`. The
  server simply doesn't surface that data on list responses (verified
  against `audiobookshelf:2.33.2`). Discovery happens through `list`;
  per-item file detail belongs on the subsequent per-item fetch.
- **`items batch-get`** — already returns ABS's expanded shape by
  default (`LibraryItemController.batchGet` always calls
  `toOldJSONExpanded()`). The CLI's `BatchGetResponse.LibraryItems`
  is `List<JsonElement>` passthrough, so callers already see the full
  data including `libraryFiles[]`. No work needed.
- **`search`** — discovery primitive, same role as `list`. Per-item
  detail belongs in a follow-up `items get --expanded` call.

## ABS endpoint touched

Same endpoint as `items get` today:

```
GET /api/items/:id
```

(`temp/audiobookshelf/server/routers/ApiRouter.js:108`, controller
`LibraryItemController.findOne` at `LibraryItemController.js:51`.)

The controller switches shape on `?expanded=1`:

```js
if (req.query.expanded == 1) {
  const item = req.libraryItem.toOldJSONExpanded()
  // ...
}
```

`toOldJSONExpanded` (`server/models/LibraryItem.js:1035-1060`)
returns every field the minified shape returns, plus:

- `oldLibraryItemId` — legacy ID for backwards compatibility (`string` or `null`)
- `lastScan` — timestamp of the last scan that touched the item (`long`, ms epoch)
- `scanVersion` — scan format version (`string`)
- `libraryFiles[]` — array of file objects (`LibraryFile.toJSON()`)
- `media` — `media.toOldJSONExpanded(id)` instead of the minified
  shape. (Most fields overlap; the expanded media has deeper
  structures.)

The minified vs expanded controller distinction is purely
client-driven by the query param. Permission, validation, and error
paths are identical.

## Command surface

```
abs-cli items get --id <itemId> [--expanded]
```

- `--id` required (unchanged).
- `--expanded` is a new boolean flag, default off. When passed, the
  CLI adds `?expanded=1` to the URL and deserializes into the new
  `LibraryItemExpanded` model.

No new permission token; this is a read endpoint and stays
auth-only.

## Output

### Default (minified) — unchanged

Same shape as today (`LibraryItemMinified`). No change to stdout.

### `--expanded` — new

The full ABS expanded shape printed verbatim. Example structure (one
seeded ebook item from the dev stack):

```json
{
  "id": "li_xyz",
  "ino": "16400000",
  "oldLibraryItemId": null,
  "libraryId": "lib_xyz",
  "folderId": "f_xyz",
  "path": "/audiobooks/Author/Title",
  "relPath": "Author/Title",
  "isFile": false,
  "mtimeMs": 1779100661814,
  "ctimeMs": 1779100661814,
  "birthtimeMs": 1779100661814,
  "addedAt": 1779100661871,
  "updatedAt": 1779100661871,
  "lastScan": 1779100662000,
  "scanVersion": "2.33.2",
  "isMissing": false,
  "isInvalid": false,
  "mediaType": "book",
  "media": { /* expanded media shape — passthrough, see below */ },
  "libraryFiles": [
    {
      "ino": "16400001",
      "metadata": {
        "filename": "multi.epub",
        "ext": ".epub",
        "path": "/audiobooks/Author/Title/multi.epub",
        "relPath": "multi.epub",
        "size": 1216,
        "mtimeMs": 1779100661814,
        "ctimeMs": 1779100661814,
        "birthtimeMs": 1779100661814
      },
      "addedAt": 1779100661871,
      "updatedAt": 1779100661871,
      "isSupplementary": false,
      "fileType": "ebook"
    },
    /* ... */
  ],
  "size": 12345
}
```

`media` stays as `JsonElement?` passthrough on the expanded model
(same as today on the minified model). The expanded `media` shape
has more depth than the minified one (e.g. expanded relationships,
fuller chapter detail), but typing it isn't part of this spec —
agents who need media internals can `jq` it.

## Error handling

No new error paths. The endpoint, validation, and permission checks
are identical to `items get`'s current behavior. The flag changes
only the response shape.

## Components

### New: `src/AbsCli/Models/LibraryItemExpanded.cs`

Three new types in this file:

- `LibraryItemExpanded` — sibling of `LibraryItemMinified`. All
  current minified fields PLUS:
  - `OldLibraryItemId` (`string?`)
  - `LastScan` (`long?`)
  - `ScanVersion` (`string?`)
  - `LibraryFiles` (`List<LibraryFile>`)
- `LibraryFile` — entry in `libraryFiles[]`. Fields:
  - `Ino` (`string`)
  - `Metadata` (`LibraryFileMetadata`)
  - `AddedAt` (`long`)
  - `UpdatedAt` (`long`)
  - `IsSupplementary` (`bool?`) — only set for ebook files; null
    otherwise
  - `FileType` (`string`) — `"ebook"` / `"audio"` / `"image"` /
    `"metadata"` / `"other"`
- `LibraryFileMetadata` — the nested `metadata` object. Fields:
  - `Filename` (`string`)
  - `Ext` (`string`) — e.g. `".epub"`
  - `Path` (`string`) — absolute server-side path
  - `RelPath` (`string`) — relative to the library folder
  - `Size` (`long`) — bytes
  - `MtimeMs`, `CtimeMs`, `BirthtimeMs` (`long`)

All three registered in `AppJsonContext` (`Models/JsonContext.cs`).
Self-test (`SelfTestCommand.cs`) gets an
`=== Expanded Item Models ===` round-trip group covering all three.

### Modified: `src/AbsCli/Services/ItemsService.cs`

One new method added alongside the existing `GetAsync`:

```csharp
public async Task<LibraryItemExpanded> GetExpandedAsync(string id)
{
    return await _client.GetAsync(
        ApiEndpoints.Item(id) + "?expanded=1",
        AppJsonContext.Default.LibraryItemExpanded);
}
```

`GetAsync(string id)` is unchanged.

### Modified: `src/AbsCli/Commands/ItemsCommand.cs`

`CreateGetCommand()` adds an `--expanded` boolean option and
branches on it:

```csharp
var expandedOption = new Option<bool>("--expanded") {
    Description = "Return the expanded shape (includes libraryFiles[], lastScan, scanVersion, oldLibraryItemId)"
};
// ... add to command's options list ...

command.SetAction(async (parseResult, cancellationToken) =>
{
    var id = parseResult.GetValue(idOption)!;
    var expanded = parseResult.GetValue(expandedOption);
    var (client, _) = CommandHelper.BuildClient();
    var service = new ItemsService(client);
    if (expanded)
    {
        var result = await service.GetExpandedAsync(id);
        ConsoleOutput.WriteJson(result, AppJsonContext.Default.LibraryItemExpanded);
    }
    else
    {
        var result = await service.GetAsync(id);
        ConsoleOutput.WriteJson(result, AppJsonContext.Default.LibraryItemMinified);
    }
    return 0;
});
```

The `--help` for `items get` adds the new flag's description and a
second response-shape block:

```
Response shape:
  { /* LibraryItemMinified */ }

Response shape (--expanded):
  { /* LibraryItemExpanded */ }
```

This will inflate the help output for one command. The v0.6.0
"extended help" follow-up (separate roadmap bullet) will hide
response-shape blocks behind an explicit flag (`--help-shape` or
similar) to address the bloat across the CLI.

### Modified: `src/AbsCli/Models/JsonContext.cs`

Three new `[JsonSerializable]` attributes for the three new types.

### Modified: `src/AbsCli/Commands/SelfTestCommand.cs`

New `=== Expanded Item Models ===` group with one `Check` per type:
`LibraryItemExpanded`, `LibraryFile`, `LibraryFileMetadata`.

### Unchanged: everything else

`ApiEndpoints.cs`, `AbsApiClient.cs`, and every other service stay as
they are. The new behavior is fully contained in the items get
command + ItemsService + new models.

## AOT considerations

Standard discipline:
- All three new types registered in `AppJsonContext` before reference.
- Serialization through `AppJsonContext.Default.<TypeName>`.
- No reflection-based overloads.

Same lane as every other v0.5.0 model addition.

## `--help` text constraint

The `items get` Caveats section gains one bullet noting the
trade-off:

- "`--expanded` returns ABS's expanded shape including `libraryFiles[]` (use it to find ebook file inos for `items toggle-ebook-status`, etc.). Default response is minified."

That's the only --help addition beyond the standard auto-generated
flag/response-shape blocks. No new permission tag (read-only,
unchanged).

### Smoke test

The seeded `Multi Ebook Test` item from the previous PR's seed
extension is the natural fixture.

New `=== Items Get Expanded ===` block in `docker/smoke-test.sh`:
1. Find the seeded multi-ebook item ID via `$CLI items list
   --library $LIB_ID --limit 100`.
2. `$CLI items get --id $ID` (default, minified) → assert response
   does NOT contain `libraryFiles`.
3. `$CLI items get --id $ID --expanded` → assert response contains
   `libraryFiles` with at least 2 entries, both with `fileType:
   "ebook"`.
4. Assert that one of the ebook entries has `isSupplementary: true`
   and one has `isSupplementary: false`.

Existing `=== Toggle Ebook Status ===` block: replace its raw
`curl ?expanded=1` calls with `$CLI items get --id $ID --expanded`
(three call sites). This is the "no more raw curl for state reads"
cleanup that motivated the PR — landing it in the same change keeps
the smoke consistent.

## Out of scope (deferred)

- **`items list --expanded`** — ABS doesn't return `libraryFiles[]`
  from the list endpoint regardless of the query param. Not useful
  to expose.
- **`items batch-get` typed expansion** — the response already flows
  through as JsonElement passthrough, agents already see
  `libraryFiles[]` in the output. Typing it would be busywork.
- **`search` flag** — discovery primitive, same role as list.
- **`?include=progress,rssfeed,downloads,share`** — orthogonal
  feature on the same endpoint. v0.6.0 roadmap.
- **Typed `media` model on the expanded shape** — agents who need
  the deeper media internals use `jq`. Typing media would be a
  separate spec (`BookMediaExpanded`, podcast variant, etc.).
- **Extended help mechanism that hides response shapes by default**
  — v0.6.0 roadmap. This PR accepts the temporary `--help` bloat.

## Open questions

None. All decisions made during brainstorming on 2026-05-13.

## v0.6.0 roadmap additions (committed alongside this PR)

Two follow-up bullets added under `### v0.6.0 — TBD`:

1. **Extended help mode** — hide response-shape blocks from `--help`
   by default; surface them via an explicit flag
   (e.g. `--help-shape`, `--help-full`, or a separate
   `items get help-shape` subcommand). The `items get --expanded`
   work adds a second response-shape block, which compounds with
   every other command's existing block; without a way to hide them,
   the help output drifts toward unreadable.
2. **`items get --include=<flags>`** — opt-in for ABS's
   `?include=progress,rssfeed,downloads,share` query parameter. Adds
   user-progress, RSS feed, download, and share state to the
   response (each as a nullable field). Works on both minified and
   expanded paths. Orthogonal to `--expanded`.

# Permission-required help section + `items toggle-ebook-status` — design

Status: approved 2026-05-13. Targets v0.5.0.

## Summary

Two related changes bundled into one spec:

1. **Permission-required help section.** A standardised top-positioned `--help` section that declares the ABS permission (`admin`, `update`, `upload`, `download`, `delete`) the calling token needs. Applied as a sweep across every existing command in the CLI; commands that need no special permission render no section.
2. **`items toggle-ebook-status`.** New command wrapping ABS's `PATCH /api/items/:id/ebook/:fileid/status` for flipping which ebook file is primary on items that hold more than one ebook format.

The two are bundled because the new command would otherwise ship without consistent permission tagging across its peers, and the sweep is much cheaper to do once than to retrofit.

## Motivation

Today's `--help` is silent about the permission an agent's token needs to invoke each command. Two commands (`encode-m4b`, `embed-metadata`) hint at it inline with `(admin-only)` in their descriptions; the other thirty-plus commands say nothing anywhere. Agents see the `permissionHint` at runtime only after a 403 — there's no way to plan ahead.

A dedicated, terse "Permission required" section makes the constraint discoverable at design time. For `canUpdate`-bound commands like the new `toggle-ebook-status`, the section is the only `--help`-visible signal that the command isn't read-only.

`items toggle-ebook-status` itself is the third of four v0.5.0 file-management primitives. The endpoint exists in ABS today; the CLI just lacks a wrapper.

## Part 1: Permission-required help section

### Helper

New extension in `src/AbsCli/Commands/HelpExtensions.cs`:

```csharp
public static void AddPermissionRequired(this Command command, string permission)
    => command.AddHelpSection("Permission required", HelpSectionPosition.Top, permission);
```

One method, one line. Reuses the existing `AddHelpSection` infrastructure — no new rendering work, no new tests for the rendering itself.

### Tokens

Match the field names in ABS's `permissions` model (`server/models/User.js`, mirrored in `docker/seed.sh`):

- `admin` — `isAdminOrUp` middleware
- `update` — `canUpdate`
- `upload` — `canUpload`
- `download` — `canDownload`
- `delete` — `canDelete`

Commands with no special permission (any authenticated user) get no section at all. The absence is meaningful: it means "any logged-in token works".

### Render

```
Permission required:
  update
```

Top of every relevant command's `--help`, between Description and Usage.

### Coverage mapping

**`admin`:**
- `backup create`, `backup list`, `backup apply`, `backup download`, `backup delete`, `backup upload`
- `items encode-m4b start`, `items encode-m4b cancel`
- `items embed-metadata`, `items batch-embed-metadata`
- `items scan`
- `libraries scan`

**`update`:**
- `items update`
- `items batch-update`
- `items chapters set`
- `items toggle-ebook-status` (this PR)
- `authors update`
- `authors match`
- `authors delete`

**`upload`:**
- `upload`
- `items cover set`
- `authors image set`

**No tag (read-only authenticated):**
- `items list`, `items get`, `items batch-get`, `items chapters lookup`, `items cover get`, `items cover remove`
- `libraries list`, `libraries get`
- `series list`, `series get`
- `authors list`, `authors get`, `authors lookup`, `authors image get`, `authors image remove`
- `metadata search`, `metadata providers`, `metadata covers`
- `tasks list`
- top-level `search`
- `login`, `config`, `changelog`, `self-test`

### Verification during implementation

Five command-to-token mappings are educated guesses and need to be confirmed against the ABS controller before tagging:

- `items scan` — likely `admin` via Tools controller; confirm.
- `libraries scan` — likely `admin`; confirm.
- `authors delete` — `update` vs `delete`. The CLI service has no `permissionHint` today; trace `AuthorController` for the actual check.
- `authors image remove` — likely `upload` (same as set) or auth-only; trace.
- `items cover remove` — same as authors image remove; trace.

For each, read the controller's `if (!req.user.canX)` line before tagging. If the controller has no per-handler check (only the top-level middleware), use the middleware's gate. Update the mapping if any disagree with the listed defaults.

**Also reconcile the runtime `permissionHint`.** If verification reveals that a service method's existing `permissionHint` is wrong, missing, or inconsistent with the ABS controller's actual check, fix it in the same change. The help-section tag and the 403 error message should agree (e.g. tag `update` ↔ hint `"'update' permission"`; tag `delete` ↔ hint `"'delete' permission"`). Concretely:
- If `authors delete` turns out to require `delete` (not `update`), set `permissionHint: "'delete' permission"` on its service call AND tag the command `delete`.
- If `items cover remove` / `authors image remove` need a `permissionHint` they don't have today, add it.
- For admin-only commands the existing convention is `"admin permission"` (no quotes around `admin`); preserve that wording rather than reformatting to `"'admin' permission"`.

## Part 2: `items toggle-ebook-status`

### Command surface

```
abs-cli items toggle-ebook-status --id <itemId> --ino <fileIno>
```

Both flags required.

### ABS endpoint

```
PATCH /api/items/:id/ebook/:fileid/status
```

Source: `temp/audiobookshelf/server/routers/ApiRouter.js:128`, controller `LibraryItemController.updateEbookFileStatus` (`LibraryItemController.js:1111-1158`).

Permission: `canUpdate` via the generic `LibraryItemController.middleware` (`LibraryItemController.js:1189`). Same bar as `items update`, `items chapters set`. NOT admin.

Body: empty PATCH. No query params. The targeted file is the entire payload, encoded as the URL parameter.

Response: `res.sendStatus(200)` — empty body. Side effects:
- `media.ebookFile` is rewritten (set to the new primary, or `null` if a primary was just unset).
- Every ebook entry in `libraryFiles` has its `isSupplementary` flag re-synced.
- `isMissing` is recomputed against `media.hasMediaFiles`.
- Socket `item_updated` broadcast.

### Toggle semantics

The endpoint flips the targeted file's `isSupplementary` value (`LibraryItemController.js:1129-1140`):

| Target file currently | Effect of one PATCH call |
|---|---|
| Supplementary | Becomes primary. The previously-primary ebook (if any) is auto-demoted in the same call. |
| Primary | Becomes supplementary. `media.ebookFile` is set to `null`. **No other file is auto-promoted.** |

Swapping epub → pdf primary = one call targeting the pdf.

### Output

CLI-synthesized receipt to stdout, exit 0. No follow-up HTTP.

```json
{
  "libraryItemId": "li_xyz",
  "fileIno": "12345678",
  "action": "toggle-ebook-status",
  "toggled": true
}
```

The receipt confirms ABS accepted the request. To inspect resulting state, callers call `items get --id <id>` and read `media.ebookFile` and `libraryFiles[].isSupplementary`. The CLI does not pre-fetch or post-fetch — thin pass-through.

### Help-text

Sections rendered:

```
Permission required:
  update

Caveats:
  Toggle: targeting a supplementary makes it primary; targeting the current primary unsets it (no auto-promote).
  --ino comes from 'items get --expanded' → libraryFiles[].ino.

Response shape:
  { "libraryItemId": "<string>", "fileIno": "<string>", "action": "<string>", "toggled": false }
```

### Error handling

The CLI uses `AbsApiClient.cs`'s existing status-code mapping with `permissionHint: "'update' permission"` on the service call.

| Status | Cause | CLI behaviour | Exit |
|---|---|---|---|
| 200 | Success | Receipt to stdout | 0 |
| 400 "Invalid ebook file id" | File ino is not an ebook (`LibraryItemController.js:1126`) | Generic 400 passthrough | 2 |
| 400 (item not a book) | `req.libraryItem.isBook` false | Generic 400 passthrough | 2 |
| 403 | Caller lacks `canUpdate` | "Permission denied. This operation requires 'update' permission." via `permissionHint` | 2 |
| 404 (item) | Item not found by middleware | Generic 404 passthrough | 2 |
| 404 (file) | "Library file '<ino>' does not exist for library item" (`:1177`) | Generic 404 passthrough — message names the bad ino | 2 |

CLI-side validation: `--id` and `--ino` are both required; System.CommandLine enforces.

### Smoke test

The seed currently uploads only audio files. To exercise toggle-ebook-status against real state, extend `docker/seed.sh` to create one library item with two ebook files (e.g. `.epub` + `.pdf`) using minimal valid fixtures generated at seed time. A bare-bones epub is a ZIP with `mimetype`, `META-INF/container.xml`, and a single XHTML chapter; a bare-bones pdf is a `%PDF-1.4`-prefixed minimal PDF. Both are generatable from Python's stdlib inside the seed script.

Smoke flow:
1. Identify the seeded multi-ebook item via `items list` filtered for it.
2. Read initial state: which file is `media.ebookFile.ino`.
3. Target the supplementary file → assert receipt; refetch and confirm it became primary (and the previously-primary became supplementary).
4. Recovery toggle: target the (now-supplementary) original-primary file → assert state restored.
5. Negative: `--id` of a different item (audio-only) → expect 400.
6. Negative: `--id` and a bogus `--ino` → expect 404.
7. Permission denial: as `readonlyuser` (no `canUpdate`) → expect `'update' permission` denial.

Existing smoke sections gain no new assertions — the help-text rendering is unit-tested.

## Out of scope (deferred)

- **Pre-fetch-and-refuse helper.** A `set-primary-ebook` verb that errors if the target is already primary would need a pre-GET; violates thin pass-through.
- **`--unset` flag.** Server has no "unset only" mode; the toggle is the mechanism.
- **Renaming the verb.** Locked to `toggle-ebook-status` after brainstorming.
- **Refactoring `permissionHint` strings to use the same tokens as the help section.** The strings are user-facing on 403 ("requires 'update' permission"); the tokens are tags. They overlap in spirit but differ in form. Consolidating is a separate cleanup.
- **Tagging permission tokens with descriptions in help.** E.g. "Permission required: update (canUpdate flag on user)" — adds noise. Single-word tag matches the seed.sh field names, which is documentation enough.

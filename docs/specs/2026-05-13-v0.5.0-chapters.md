# `items chapters` — design

Status: approved 2026-05-13. Targets v0.5.0.

## Summary

Add two leaf commands under `items chapters` that mirror the two ABS
REST endpoints for chapter metadata — one for an Audnexus-backed
read-only probe, one for writing chapters onto a library item:

```
abs-cli items chapters lookup --asin <asin> [--region <r>]
abs-cli items chapters set    --id <id> {--input <file> | --stdin}
```

Both endpoints exist in ABS today; the CLI is a thin pass-through.
There is intentionally no third `get` verb — `media.chapters` is
already a field on `GET /api/items/:id`, not a dedicated chapter
endpoint, and projecting it through the CLI would not be 1:1 with
REST.

## Motivation

Agents fixing chapter metadata for an item have two natural inputs:
the canonical Audnexus chapter list keyed by Audible ASIN, or a
hand-curated chapter array produced by some other tool. The first
needs a lookup primitive; both need a writer that hits
`POST /api/items/:id/chapters`. The current CLI has neither, so
agents either curl the endpoint themselves or skip chapter cleanup.

Second of four v0.5.0 "file management" primitives. Pairs with the
later `items embed-metadata` (separate spec) for the case where the
caller wants the chapter timings inside the audio file, not just in
ABS's database and sidecar.

## ABS endpoints touched

| CLI verb | Endpoint | Method | Body | Notes |
|----------|----------|--------|------|-------|
| `lookup` | `/api/search/chapters` | GET | (none) | Query params: `asin` (required), `region` (optional, server default `us`). Returns Audnexus shape on success; HTTP 200 with `{error, stringKey}` on no match, invalid ASIN, or invalid region. |
| `set` | `/api/items/:id/chapters` | POST | `{ chapters: [{title, start, end}, ...] }` | Updates ABS DB + sidecar. Does **not** touch the audio file. Returns `{success, updated}`. |

Routes confirmed at
`temp/audiobookshelf/server/routers/ApiRouter.js:122` (`set`) and
the search router for `lookup`. Controllers:
`temp/audiobookshelf/server/controllers/SearchController.js:186`
(`findChapters`) and
`temp/audiobookshelf/server/controllers/LibraryItemController.js:850`
(`updateMediaChapters`).

Permissions:
- `lookup` — standard authenticated user. No admin gate.
- `set` — `canUpdate`. Same level as `items update`. Not admin.

## Command surface

```
items chapters lookup --asin <asin> [--region <r>]
items chapters set    --id <itemId> {--input <file> | --stdin}
```

### `lookup`

- `--asin` required. CLI does not regex-validate before sending;
  ABS's `isValidASIN` runs server-side and returns an in-band error
  body on failure (see "Error handling").
- `--region` optional. Omitted from the URL when the caller doesn't
  pass it; ABS applies its default `us`. Same shape as `authors
  match --region`.

### `set`

- `--id` required.
- Exactly one of `--input <file>` and `--stdin` required. Mirrors
  `items batch-update` / `items batch-get`. No inline JSON support
  — chapter arrays are too long to fit on a command line, and
  shipping yet another precedent variant on top of `items update`'s
  string-or-path `--input` is not worth it (a separate roadmap
  bullet for v0.6.0 brings `items update` in line with the
  batch-* shape).
- Input is the full ABS request body
  `{ "chapters": [{ "title": "...", "start": 0, "end": 12.3 }, ...] }`
  in **seconds**. The CLI does **not** convert from Audnexus's
  millisecond shape — that conversion belongs to the agent (see
  "Out of scope").

## Validation

Input validation on `set` is performed by AOT JSON deserialization
into a typed `ChaptersSetRequest` model — the same shape ABS itself
checks (`title` string, `start` number, `end` number, body is
`{chapters: [...]}`). No semantic checks: the CLI does not enforce
`end > start` and does not check for overlapping chapters, matching
ABS's own behavior. A malformed body fails at deserialization with
exit 1 and a stderr message naming the bad field; valid-shape but
semantically-broken arrays are sent and ABS stores them as given.

## Output

### `lookup` — happy path

Audnexus response passed through verbatim to stdout, exit 0. The
shape is millisecond-based:

```json
{
  "asin": "B0XXXXXXX",
  "brandIntroDurationMs": 0,
  "brandOutroDurationMs": 0,
  "chapters": [
    { "title": "Opening Credits", "lengthMs": 12345, "startOffsetMs": 0, "startOffsetSec": 0 }
  ],
  "isAccurate": true,
  "runtimeLengthMs": 0,
  "runtimeLengthSec": 0
}
```

The CLI does no projection or transformation. `isAccurate` and the
runtime fields are how agents decide whether to apply the result.

### `set` — happy path

ABS response passed through verbatim to stdout, exit 0:

```json
{ "success": true, "updated": true }
```

`updated` is `false` when the posted chapters match the existing
array byte-for-byte (same length, same `title`/`start`/`end` in
order). Agents can rely on this as a "did anything change" signal.

### Failure paths

See "Error handling" below.

## Error handling

### `lookup` — in-band errors on HTTP 200

ABS's `SearchController.findChapters`
(`SearchController.js:194-210`) returns HTTP 200 with a top-level
`error` key for three failure modes. The CLI inspects every 200
response from this endpoint and translates any body containing an
`error` key into exit 2 with the error string on stderr:

| Server condition | Response body (HTTP 200) | CLI behaviour | Exit |
|---|---|---|---|
| No Audnexus match | `{"error":"Chapters not found","stringKey":"MessageChaptersNotFound"}` | stderr `"Not found. Chapters not found"` | 2 |
| Bad ASIN | `{"error":"Invalid ASIN","stringKey":"MessageInvalidAsin"}` | stderr `"Invalid ASIN"` | 2 |
| Bad region | `{"error":"Invalid region","stringKey":"MessageInvalidRegion"}` | stderr `"Invalid region"` | 2 |
| Internal Audnexus error | HTTP 500 `{"error":"Internal server error"}` | Generic 5xx passthrough | 2 |

Detection rule: deserialize into a discriminated wrapper that holds
either the `ChaptersLookupResponse` (success) or
`ChaptersLookupError` (in-band failure) variant. The presence of an
`error` field on the response is the discriminator. On the error
variant, exit 2 and write `error` to stderr prefixed with the
contextually appropriate `Not found.` for the "Chapters not found"
case to match the `authors match` precedent; other in-band errors
surface the raw `error` string.

### `set` — standard status mapping

The CLI relies on `AbsApiClient.cs`'s existing status-code mapping;
no new behavior required.

| Status | Cause | CLI behaviour | Exit |
|--------|-------|---------------|------|
| 200 | Success | `{success, updated}` to stdout | 0 |
| 400 | Body shape invalid per ABS check (`LibraryItemController.js:861`) | Generic 400 passthrough | 2 |
| 403 | Caller lacks `canUpdate` | stderr `"Permission denied. This operation requires 'update' permission."` via `permissionHint` | 2 |
| 404 | Item ID does not exist (handled by route middleware before the controller) | Generic 404 passthrough | 2 |
| **500** | Item missing / not a book / no audio tracks (`LibraryItemController.js:856-858` — non-conventional, ABS returns 500 instead of 400/404 here) | Generic 5xx passthrough | 2 |

The 500-on-invalid-item case is documented in `--help` so agents
seeing a 500 from `set` know to check item state before assuming a
server fault.

CLI-side validation (before any HTTP call):
- `--id` required (`set`).
- `--asin` required (`lookup`).
- Exactly one of `--input` / `--stdin` (`set`); explicit error if
  both or neither.
- Body deserializes into `ChaptersSetRequest` — failures rejected
  with exit 1 before HTTP.

## Components

### New: `src/AbsCli/Services/ChaptersService.cs`

Static-instance service following the project's per-domain pattern.
Two methods:

```csharp
public Task<ChaptersLookupResult> LookupAsync(string asin, string? region);
public Task<ChaptersSetResponse> SetAsync(string itemId, string jsonBody);
```

`LookupAsync` GETs the search endpoint, deserializes into a wrapper
that distinguishes success from in-band error (see
`ChaptersLookupResult` below), and returns the result. `SetAsync`
POSTs the caller's pre-validated body through the standard
`AbsApiClient.PostAsync` path with `"'update' permission"` as the
`permissionHint`.

### New: `src/AbsCli/Models/ChapterModels.cs`

Six model types:

- `ChaptersLookupResponse` — Audnexus success shape. Fields:
  `Asin`, `BrandIntroDurationMs`, `BrandOutroDurationMs`,
  `Chapters: List<AudnexusChapter>`, `IsAccurate`,
  `RuntimeLengthMs`, `RuntimeLengthSec`.
- `AudnexusChapter` — `{ Title, LengthMs, StartOffsetMs, StartOffsetSec }`.
- `ChaptersLookupError` — `{ Error, StringKey }`. Used to detect
  and surface the in-band error case.
- `ChaptersLookupResult` — CLI-internal discriminated wrapper
  `{ Success: ChaptersLookupResponse?, Error: ChaptersLookupError? }`.
  Not serialized; just a return-type carrier between service and
  command.
- `ChaptersSetRequest` — `{ Chapters: List<ChapterWriteEntry> }`.
  Used both as the deserialization target for `--input`/`--stdin`
  (validating input shape) and as the request body sent to ABS.
- `ChapterWriteEntry` — `{ Title, Start, End }`. Nested inside
  `ChaptersSetRequest` but registered separately in
  `AppJsonContext` for source-generated serialization.
- `ChaptersSetResponse` — `{ Success, Updated }`.

All serializable types registered in `AppJsonContext`
(`Models/JsonContext.cs`); `ChaptersLookupResult` is internal-only
(not over the wire) so it is not registered. Self-test
(`SelfTestCommand.cs`) adds a `=== Chapter Models ===` round-trip
group covering every registered type.

### New: `ItemsCommand.CreateChaptersCommand()`

Method added to `src/AbsCli/Commands/ItemsCommand.cs`. Returns a
`Command("chapters", "...")` with two subcommands (`lookup`, `set`)
wired in. Registered as a subcommand of `items` alongside the
existing `list`, `get`, `update`, `cover`, `encode-m4b`, etc.

Per project convention: each subcommand has its own `AddExamples`
and an `AddResponseExample<T>` pointing at the relevant success
shape.

### Modified: `src/AbsCli/Api/ApiEndpoints.cs`

Two constants added:

```csharp
public const string SearchChapters = "/api/search/chapters";
public const string ItemChapters   = "/api/items/{0}/chapters"; // POST
```

`AbsApiClient` and `ItemsService` already do parameterized URL
construction in this style.

## AOT considerations

Project ships as Native AOT (`PublishAot=true`). Standard
discipline applies:

- **No reflection-based JSON.** All new models registered in
  `AppJsonContext`; serialization through
  `AppJsonContext.Default.<TypeName>`.
- **Discriminator parse:** `ChaptersLookupResult` is built in
  service code by attempting deserialization into
  `ChaptersLookupError` first (cheap — only two known keys); on a
  match it's the error variant, otherwise re-parse as
  `ChaptersLookupResponse`. Both source-generated. No
  `JsonDocument` reflection path.
- **No System.CommandLine handler reflection.** Both leaf commands
  use the existing `SetAction(async (parseResult, ct) => …)` style
  already in the codebase.

Nothing new about this work is AOT-risky; same lane as `metadata
search`.

## `--help` text constraint

Calibrate against the leaner existing commands (e.g. `items scan`,
`authors match`). Terse one-liners, no prose blocks. Caveats from
above surface as the shortest accurate sentence. Caveats that must
appear in `--help` somewhere on the verb tree:

**`items chapters lookup`:**
- Audnexus-backed; same backing service as `authors lookup`/`match`.
- Returns ms-based shape; `items chapters set` takes seconds. No
  CLI conversion.
- No upstream match / invalid ASIN / invalid region → exit 2 with
  the ABS error string on stderr.

**`items chapters set`:**
- Body must be `{chapters: [{title, start, end}, ...]}` in seconds.
- Writes ABS DB and sidecar only. Audio files unchanged — use the
  planned `items embed-metadata` to bake chapters into the file
  itself.
- ABS server returns 500 (not 400/404) when the item is missing,
  not a book, or has no audio tracks.
- No semantic validation: `end > start` and non-overlap are not
  checked by ABS or by the CLI.

Wording calibration happens at implementation time, not in this
spec.

## Testing

### Unit tests (`tests/AbsCli.Tests`)

- `lookup`: in-band error variants (`MessageChaptersNotFound`,
  `MessageInvalidAsin`, `MessageInvalidRegion`) each produce exit 2
  with the right stderr; success body passes through to stdout.
- `lookup`: `--region` omitted from URL when caller doesn't pass it;
  included when passed.
- `set`: malformed JSON (missing `title`, wrong type for `start`,
  not wrapped in `{chapters: [...]}`) rejected with exit 1 before
  any HTTP call.
- `set`: `--input` and `--stdin` mutual exclusion; explicit error
  when both or neither given.
- Self-test (`SelfTestCommand`) round-trip group for all five
  serializable model types.

### Smoke test (`docker/smoke-test.sh`)

**`set` (runs unconditionally):**

1. Seed produces a known library item with audio tracks. Take its
   ID and call `abs-cli items get --id $ID` to read the existing
   `media.chapters` (could be empty).
2. Write a hand-crafted chapters JSON to a tempfile:
   `{"chapters":[{"title":"Test Chapter 1","start":0,"end":1.5},
   {"title":"Test Chapter 2","start":1.5,"end":3.0}]}`
3. `abs-cli items chapters set --id $ID --input <tempfile>` →
   assert `{"success":true,"updated":true}`.
4. Repeat the same `set` → assert `{"success":true,"updated":false}`
   (idempotence check matching the server's behavior at
   `LibraryItemController.js:869-889`).
5. `abs-cli items get --id $ID | jq .media.chapters` → assert two
   entries with the expected titles and offsets.

**`set` negative paths:**

- 400 on bad shape: pipe `{"chapters":[{"title":"x"}]}` (no
  `start`/`end`) via `--stdin` → exit 2, stderr passthrough.
- 500 on missing item: `--id li_nonexistent` → exit 2 (documented
  ABS quirk, not a CLI bug).
- 403 as `uploaduser`: `abs-cli items chapters set --id $ID
  --input <file>` exits 2 with the `'update' permission`
  permission-denied message.

**`lookup` (gated by `SMOKE_TEST_EXTERNAL=1`):**

Same gate as the existing external-provider tests at
`smoke-test.sh:764`. When enabled:

- `abs-cli items chapters lookup --asin <known-good-ASIN>` →
  exit 0, stdout has `asin`, `chapters` array, and `isAccurate`.
- `abs-cli items chapters lookup --asin B000000000` (well-formed
  but no Audnexus data) → exit 2, stderr contains
  `"Chapters not found"`.
- `abs-cli items chapters lookup --asin not-an-asin` → exit 2,
  stderr contains `"Invalid ASIN"`.

Pick the known-good ASIN from a long-tail Audible audiobook unlikely
to disappear (e.g. a public-domain classic). Document the choice in
the smoke script with a comment so a future fixture rotation has
context.

## Out of scope (deferred)

These were considered and explicitly rejected for v0.5.0. Each can
be picked up in a future spec if a real workflow demands it.

- **CLI-side unit conversion from ms to seconds.** Tempting because
  the lookup→set pipeline always needs it, but baking it in
  contradicts the thin pass-through principle and hides the
  `isAccurate` / `runtimeLengthSec` checks an agent should be doing
  between lookup and set. Document the conversion formula in
  `--help` and the README "agent workflows" section.
- **`--apply` mode that does lookup → translate → set in one
  call.** Same reasoning. The agent inspects the lookup result
  before committing; auto-apply hides that.
- **`items chapters get` projection over `items get`.** Not 1:1
  with REST. `abs-cli items get --id X | jq .media.chapters`
  covers the use case.
- **ASIN lookup by title/author.** Out of scope; `metadata search`
  already resolves ASINs via the configured providers.
- **Semantic validation** (`end > start`, non-overlap, monotonic
  ordering). ABS doesn't validate these; neither does the CLI.
  Validation at the CLI boundary would be the only place in the
  codebase where the CLI does semantic checks the server doesn't,
  and is best left to the agent (which has the source data anyway).
- **Inline JSON for `set --input`.** Not practical for chapter
  arrays; the `items update` precedent for inline-or-file `--input`
  is being retired (see roadmap v0.6.0 bullet) rather than
  expanded.
- **Embedding chapters into the audio file.** Separate v0.5.0
  bullet: `items embed-metadata` wraps the admin-only
  `POST /api/tools/item/:id/embed-metadata?forceEmbedChapters=1`
  endpoint that runs an ffmpeg pass to bake the ABS-side state
  into the audio file.
